# 小說寫作規範

**目的**：確保每一章的輸出內容可以**直接複製貼上到小說連載網站**，無需二次格式化；同時確保與世界觀設定完全一致。

> **資料來源**：所有設定相關的內容，以 `世界觀/` 資料夾中的文件為唯一正典。
> 寫作前必須參照對應的設定檔案，不得憑空捏造或與設定矛盾。

---

## 1. 核心原則

1. **純文本優先**：輸出即成品，讀者看到的就是最終版本。
2. **設定為王**：一切名詞、能力、境界、角色關係皆以 `世界觀/` 資料夾為準。
3. **沉浸感至上**：減少說明性旁白，以描寫、動作、對話推進敘事。

---

## 2. 文件與命名

| 項目 | 規範 |
|------|------|
| **章節檔名** | `chapter_XXX.md`（三位數字，例如 `chapter_001.md`） |
| **卷冊目錄** | `稿件/第N卷/`（例如 `稿件/第1卷/`） |
| **世界觀檔案** | 存放於 `世界觀/`，以編號+名稱命名（例如 `01_宇宙觀與歷史.md`） |
| **劇情大綱** | 存放於 `規劃/劇情/`（例如 `第一卷大綱.md`） |

---

## 3. 排版與格式

### 3.1 嚴禁 Markdown 格式

章節正文中**絕對不可**出現以下語法：

- ❌ `**粗體**`
- ❌ `# 標題`
- ❌ `* 列表` / `- 列表`
- ❌ `> 引用`
- ❌ `[連結](url)`

### 3.2 段落與分段

- 段落之間**空一行**。
- 對話**單獨成段**，不與敘述混在同一段。
- 場景轉換時使用**空一行**即可，不使用任何分隔符號（如 `---` 或 `***`）。

### 3.3 章節結構

```
第一行：章節標題（例如：第十六章 投名狀）
第二行：空行
第三行起：正文

……

最後一行：（本章完）
```

- **章節標題格式**：`第X章 副標題`（用中文數字，加空格後接副標題）。
- **結尾標註**：每章最後一行固定為 `（本章完）`。

---

## 4. 對話與標點

### 4.1 對話格式

- 使用**中文直角引號**：「對話內容」
- 對話中的引用或強調使用：『內引號內容』
- 角色內心獨白（非出聲）直接寫，不加引號，以思維描寫帶入。

### 4.2 標點規範

| 場景 | 使用 | 範例 |
|------|------|------|
| 一般對話 | 「」 | 「我知道了。」 |
| 引用中的引用 | 『』 | 「他說過『不要靠近後山』。」 |
| 省略/停頓 | …… | 「你是說……那個東西？」 |
| 破折號 | —— | 他看到了——一隻巨大的眼睛。 |
| 嘆號/問號 | 全角 | 「什麼！？」 |

---

## 5. 系統介面描寫

本作核心賣點之一是「賽博系統」的視覺感。系統相關的顯示必須與正文明確區分。

### 5.1 系統通知（主要提示）

使用**全角方括號** `【】` 包裹，獨立成行：

```
【系統重啟中...】
【警告：檢測到嚴重邏輯錯誤。】
【警告：宿主精神狀態危急 (理智值: 3/100)。】
```

### 5.2 系統標籤（物品/目標掃描）

使用 `【】` 包裹，可連續多行：

```
【物品：殘缺的高蛋白有機體】
【狀態：已死亡】
【危險度：無】
```

### 5.3 系統語音/文字輸出

系統以語音或簡短句子回應時，**不加引號**，直接用敘述帶入：

```
系統冰冷的聲音在腦海中響起：「雜訊已屏蔽。正在進行環境掃描。」
```

### 5.4 禁止事項

- ❌ 不使用 `<code>` 或反引號標記代碼。
- ❌ 不使用英文方括號 `[]` 替代 `【】`。
- ❌ 不使用 HTML 標籤或任何程式語法格式。

---

## 6. 字數要求

- 每章正文（含標點符號）需達到 **3000 中文字以上**。
- 若低於 3000 字，須擴充場景描寫、心理刻畫、或對話層次，**不可灌水**。

---

### 7.1 必須參照的設定檔案

寫作前，**務必查閱**以下資料：

|------|----------|

| `世界觀/02_勢力與社會.md` | 門派名稱、勢力關係、陣營劃分 |
| `世界觀/03_長生宗詳設.md` | 宗門結構、地點名稱、天樞協議、資訊階層 |
| `世界觀/04_修煉與術法.md` | 修煉等級、術法名稱、丹藥物品 |
| `世界觀/05_賽博系統.md` | 系統模組名稱與版本、能量數值、特殊協議 |
| `世界觀/06_角色檔案.md` | 角色姓名、境界、裝備、當前狀態 |

### 7.2 專有名詞一致性

以下為高頻出現的核心名詞，**絕對不可寫錯或用其他稱呼替代**：

| 類別 | 正確用詞 | 常見錯誤 |
|------|----------|----------|
| 系統 | 內核 | ❌ 系統核心、核心程式 |
| 系統介面 | 【】包裹 | ❌ []、〔〕 |
| 主角系統能力 | 解析之眼、防火牆、駭入協議、編譯器、生物反應爐 | ❌ 掃描眼、分析眼 |
| 修煉真相 | 深淵孢子、鎮魂釘、深淵母神 | ❌ 靈氣（設定中靈氣是假象） |
| 境界名稱 | 練氣期、築基期、金丹期、元嬰期 | ❌ 煉氣期（注意是「練」不是「煉」） |
| 長生宗通訊 | 天樞協議 | ❌ 通訊協議、宗門網絡 |
| 主角武裝 | Mark-1「潛淵者」 | ❌ 外骨骼、機甲 |
| 清理者 | 清理者 | ❌ 守衛傀儡、機械兵 |

### 7.3 視角與認知規則

這是本作最容易出錯的地方。由於「鎮魂釘」的存在，不同角色看到的世界不同：

- **普通修士視角**：看到的是「靈氣」「道基」「金丹」——一切正常的修仙世界。
- **主角視角**（系統過濾後）：看到的是「深淵孢子」「異體器官」「肉體裂變」——真實的恐怖世界，但被系統的馬賽克/數據流覆蓋。
- **主角視角**（系統關閉時）：直面不可名狀的恐怖，理智值暴跌。

> 寫作時必須明確當前是「誰的視角」，不可混用認知。
> 例如：普通修士不會說「深淵孢子」，他們只知道「靈氣」。

## 8. 寫作風格

### 8.1 敘事節奏

- **開場要快**：每章前 300 字內必須有鉤子（衝突、懸念、資訊落差）。
- **結尾留鉤**：每章結尾製造懸念，引導讀者翻下一章。
- **張弛有度**：高強度戰鬥/恐怖場景後，安排短暫喘息（升級、情報整理、角色互動）。

### 8.2 描寫原則

- **感官優先**：多用視覺、聽覺、嗅覺、觸覺描寫，少用「他感到害怕」這類直述。
- **具象比喻**：用讀者熟悉的事物比喻陌生概念（例：「就像是 Windows 開機的提示音」）。
- **賽博與修仙交織**：刻意製造科技詞彙與修仙語境的衝撞感（例：「防火牆」「編譯」「代碼」出現在修仙場景中）。

### 8.3 系統描寫原則

- 系統資訊要**精簡有力**，不堆砌無意義數據。
- 除非是關鍵爽點（如：大幅升級、新技能解鎖），否則不要大段列出系統面板。
- 系統的「機械感」要與角色的「人性」形成對比——系統越冷靜，場景越緊張。

### 8.4 恐怖描寫原則

- 克蘇魯式恐怖的核心是「未知」與「不可理解」，不是血漿量。
- 描寫異變時，聚焦「差一點就正常」的違和感，而非直接的獵奇。
- 主角的恐懼要有層次：從生理恐懼 → 認知恐懼 → 存在恐懼。

## 9. AI 協作規範

以下為 AI 輔助寫作時的固定指令：

### 9.1 輸出格式

- 輸出章節時，**直接輸出純文本**，不包裹在程式碼區塊中。
- 第一行為章節標題，最後一行為 `（本章完）`。
- 不在章節正文中插入任何 AI 註解或後設說明。

### 9.2 寫作前檢查清單

每次撰寫新章節前，必須確認：

- [ ] 已讀取對應的卷冊大綱
- [ ] 已查閱相關角色的最新狀態（`06_角色檔案.md`）
- [ ] 已確認主角當前境界、能量值、系統版本（`05_賽博系統.md`）
- [ ] 已確認故事發生的地點與環境（`01_宇宙觀與歷史.md`、`03_長生宗詳設.md`）
- [ ] 上一章的結尾鉤子已銜接

### 9.3 完成後檢查清單

章節完成後，必須自查：

- [ ] 字數是否達到 3000 字以上
- [ ] 是否包含任何 Markdown 語法（不可有）
- [ ] 系統提示是否統一使用【】
- [ ] 對話是否統一使用「」
- [ ] 專有名詞是否與設定一致
- [ ] 視角認知是否正確（未混用不同角色的世界觀認知）
- [ ] 章節是否以 `（本章完）` 結尾
